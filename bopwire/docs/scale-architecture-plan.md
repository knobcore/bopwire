# BOPWIRE — PHASED IMPLEMENTATION PLAN (Model 1, forge-hardened, batched, prunable)

## Migration decision (definitive)

**WIPE + re-bootstrap on the FINAL block/tx format, exactly once, at the end of Phase 4.** Rationale, decisive:
- Genesis has **no premine** (`genesis_writer.cpp` mints only the founder GRANT, no MintTx). The ~800M "founder" balance is **non-consensus per-node derived state** (direct `apply_mint`+`db_.write` at server.cpp:1061/1081) that `clear_derived_state` already destroys on any rebuild and that was never replicated. It has zero real value → **abandon it** (founder resets to ~0, which is correct).
- Phases 2–4 are consensus-breaking (new TxType 0x70, `state_root` in header, `BLOCK_VERSION 3→4`, `MAX_BLOCK_SIZE 2→8 MiB`). On a live chain these demand permanent activation-height cruft. There are **zero real users**, so the flag day is free now and never gets cheaper.
- Therefore: build+prove Phases 1–4 on throwaway local chains; the live VPS wipe (`wipe-and-run-home.ps1` + fresh `genesis_writer` on final format) happens **once, last**, strictly **before any second independent operator or public onboarding**. If a real user onboards first, the calculus flips to in-place+activation-heights — so onboarding is **blocked until after cutover**.

**Total-supply authority post-cutover:** `clear_derived_state` / block replay is the ONLY writer of `c:total_supply`. Any lingering off-chain supply write is a release-blocker.

## Phase ordering and consensus-criticality

| Phase | Title | Consensus-critical? | Adversarial pass before deploy? |
|---|---|---|---|
| 0 | Determinism prerequisites (integer burn, state-set audit, finality cap, submit_ms bounds) | **YES** | **YES** |
| 1 | Reward durability: on-chain forge-verified per-play MINT | **YES** | **YES** |
| 2 | Sybil/finality base: permissioned `v:` + signed checkpoints + fork-weight redefinition | **YES** | **YES** |
| 3 | Batched settlement (SETTLEMENT_MINT, recompute-authoritative) | **YES** | **YES** |
| 4 | state_root + snapshots + pruning (succession) | **YES** | **YES** |
| 5 | Two-tier mempool + verify-once + capped inclusion (throughput) | **YES** (block validity caps) | **YES** |

Every phase here touches block validity or the mint math, so **all six get an adversarial verification pass**. Phases are ordered so each is independently shippable to the (single) node and each later phase depends only on earlier ones.

# PHASE 0 — Determinism prerequisites

Purpose: make the arithmetic and reorg surface deterministic **before** any of it becomes state-root-load-bearing. No wire change; in-place; reversible.

### Edits
- `src/tokens/ledger.cpp : compute_burn_rate (10-21)` : **replace `double` cubic with integer fixed-point.** Compute `num = (total_supply - SUPPLY_FLOOR)`, `den = (SUPPLY_CAP - SUPPLY_FLOOR)`; `burn = muldiv_u128(muldiv_u128(num, num, den), num, den) * 1000 * TOKEN_DECIMALS / den` using a `__uint128_t` (or portable 128 helper) `muldiv` that never touches floating point. Keep the `<SUPPLY_FLOOR→0` and `>=SUPPLY_CAP→uint64_max` branches. (H1)
- `src/consensus/candidate.h : alongside PROPAGATION_WINDOW_MS (44)` : add `FINALITY_DEPTH`, `MAX_SUBMIT_SKEW_MS`. Pick `FINALITY_DEPTH` = a few hours of blocks; `MAX_SUBMIT_SKEW_MS` small (e.g. 5000).
- `src/core/chain.cpp : reorg_to_branch (1159 area)` : **enforce reorg depth cap** — reject any branch whose `fork_height < tip_.height - FINALITY_DEPTH`. This makes pruning below `FINALITY_DEPTH` safe (H2/M/C1 prerequisite).
- `src/api/rats_api.cpp : ingest_tx / tx_preflight_ok (3347, 3331)` and `src/core/chain.cpp : apply block (217)` : **bound `submit_ms`** — reject any tx whose `submit_ms < tip.block_ts - PROPAGATION_WINDOW_MS` (backdated-mature) or `> tip.block_ts + MAX_SUBMIT_SKEW_MS` (future). Cap min-merge re-flood to **one lowering per tx_hash** (record `pt:` already-lowered flag). (H3)
- `src/storage/database.cpp : clear_derived_state (~990) + rebuild_derived_state` : **AUDIT + FIX** so rebuild reconstructs EVERY block-derived prefix, not just `a:/s:/f:/i:/u:/bh:`. Must also rebuild `nv:` (nonces), `c:total_supply`, `v:`, governance (`founder:/mlvl:/mpub:/mact:/m:/slashed:`), names (`un:/addrun:/label:/art_label:`), proposals (`prop:/propstatus:/propvote:`), moderation (`ha:/hb:/ht:/d:/fr:/ml:/ms:`). (M4 — prerequisite for Phase 4 state_root.)

### Consensus predicate (preflight == block-apply)
```
valid_tx_timing(tx, tip):
  tip.block_ts - PROPAGATION_WINDOW_MS <= tx.submit_ms <= tip.block_ts + MAX_SUBMIT_SKEW_MS
```
`compute_burn_rate` must return **bit-identical** values on x86/ARM, any `-O`, any `-ffast-math`.

### Verification checklist
- [ ] Cross-arch test vector: 10k `total_supply` samples → identical burn on x86-64 and ARM64 builds.
- [ ] Unit test: reorg deeper than `FINALITY_DEPTH` is rejected; at exactly `FINALITY_DEPTH` accepted.
- [ ] Backdated `submit_ms` tx rejected identically in preflight and apply.
- [ ] `rebuild_derived_state` on a populated chain yields byte-identical LevelDB (all listed prefixes) vs a live-built node — diff every prefix.

# PHASE 1 — Reward durability: on-chain forge-verified per-play MINT

Purpose: play rewards become **replicated, forge-verified transactions** instead of per-node direct writes. This is the durability beachhead — players earn balances that survive resync and match across nodes. In-place, reversible (revert = restore direct write).

### Edits
- `src/api/server.cpp : session.complete (1044-1086)` : **remove `chain_.apply_mint` + `db_.write`.** Keep the PlayProof build+sign (990-1032) and all anti-farm gates. Wrap the MintTx via `Transaction::from_mint` (transaction.cpp:540 — gains its **first live caller**), stamp origin `submit_ms` **once**, submit via `ingest_tx` (MC_TX_TYPE flood + min-merge). Keep `ddur:`/`ddaymax:` as **node-local, non-consensus** rows — write them in a **separate** batch, NOT with any consensus write. (Fixes CURRENT REWARD STATE; M3.)
- `src/tokens/mint.h/.cpp : refactor validate_mint (129-174)` : extract `check_play(proof, db, err)` (node_sig vs `v:` registry, session-unused, `duration>=30000`, heartbeat plausibility) as a reusable predicate. Add `recompute_mint(proof, db) -> {outputs, burn}` wrapping `compute_mint_outputs` with the on-chain SongSection + play_count + serving-node wallet resolved from `v:` (see H6 binding, finalized in Phase 2).
- `src/api/rats_api.cpp : tx_preflight_ok MINT case (3331)` : **replace bare deserialize** with `check_play(...)` **AND** `recompute_mint(...)`; reject if declared outputs/burn ≠ recompute. A forged/mismatched mint never floods.
- `src/core/chain.cpp : apply block MINT branch (217-225)` : run the **identical** `check_play` + `recompute_mint` + equality gate **before** `apply_mint`, so a block carrying a forged/mismatched mint **fails to connect**.

### Wire format
No new type. MintTx unchanged; only its **validation** is added and its transport path (flood) is now exercised.

### Consensus predicate (preflight == block-apply)
```
valid_mint(mint, db):
  check_play(mint.proof, db)                       # node sig ∈ v:, session-unused, dur>=30s, hb plausible
  AND recompute_mint(mint.proof, db) == {mint.outputs, mint.burn}   # byte-identical
  AND valid_tx_timing(mint, tip)                   # Phase 0
```

### Verification checklist
- [ ] Player balance survives a full `clear_derived_state` rebuild and matches on a second node (the core defect closed).
- [ ] A hand-forged MintTx with tampered outputs is rejected in BOTH preflight and apply (same error).
- [ ] `ddur:`/`ddaymax:` never appear in a batch that also writes `a:`/`c:`/`s:`.
- [ ] Two nodes fed the same MintTx flood produce byte-identical blocks.

# PHASE 2 — Sybil / finality base (the load-bearing security phase)

Purpose: without this, forging a play is **both free money and free chain-takeover** (C1/C3), and migrating away from per-play MINT zeroes honest fork weight (C2). This phase must land **before** batching or snapshots are trusted.

### 2A — Permission the `v:` registry (C3)
- `src/api/rats_api.cpp : v: write path` : node registration (`v:`+serving_node_id → pubkey33+endpoint) becomes **permissioned**: a `v:` write is only accepted if authorized by a founder/mod key (`founder:`/`mpub:`, already in state) via a signed `NodeAuthTx`, and is **slashable** (reuse `slashed:` + SlashTx). Choose permissioned-registry over proof-of-listener for v1 (proof-of-listener is L3 follow-up). This makes "who can mint" a closed, accountable set.
- Optional hardening (schedule as follow-up, not a Phase-2 blocker): **proof-of-listener** — plays carry a listener-device signature so a node cannot unilaterally fabricate the counterparty. Track as L3.

### 2B — Signed checkpoints + reorg pinning (C1/C5)
- `src/core/chain.cpp : new CheckpointTx apply + checkpoints_ map (reuse add_config_checkpoints:30, checkpoint_ok, reorg_to_branch:1156)` : every `K` blocks a founder/mod key signs `{height, block_hash}` and floods a `CheckpointTx`; on apply it writes `checkpoints_[height]=hash`. `reorg_to_branch` **already** rejects branches violating checkpoints (verified 1156-1161) — now checkpoints are dynamic and signed, not just genesis. A syncing/succession node **refuses any header chain or snapshot not descending from a checkpoint newer than the snapshot height** (closes C5).
- Hardcode/rotate the latest checkpoint in the binary at release time.

### 2C — Redefine fork weight so it is unforgeable and batch-stable (C2)
- `src/core/chain.cpp : count_plays (41-46)` : redefine block weight as **verified constituents** = for a MINT, 1; for a SETTLEMENT_MINT (Phase 3), the **verified constituent_count** (post-validation), **hard-capped** at `MAX_CONSTITUENTS_PER_SETTLEMENT`. This keeps weight continuous across the MINT→SETTLEMENT migration (C2) and, combined with the constituent cap + signed checkpoints, removes the "declare huge count for free weight" attack. Between the honest-checkpointed tip, weight is a tiebreak only within `FINALITY_DEPTH`.

### 2D — Bind serving-node wallet (H6)
- `src/core/transaction.h/.cpp` + `v:` value : **commit the serving-node wallet address** inside the mint/settlement under the node signature, OR enforce the invariant `serving_node_address == address(v:pubkey)`. Pick the invariant (smaller, one bound source). `recompute_mint` resolves the wallet from that single source; a mint whose implied wallet ≠ bound wallet is rejected.

### Consensus predicate additions
```
valid_checkpoint(cp): sig(cp) verifies against a founder/mpub key in state
weight(block): sum over txs of
    MINT -> 1
    SETTLEMENT_MINT -> min(verified_constituents, MAX_CONSTITUENTS_PER_SETTLEMENT)
    else -> 0
sync_admissible(header_chain, snapshot_h):
    exists checkpoint C with C.height >= snapshot_h AND header_chain descends from C
v: write admissible iff authorized by founder/mod signature
```

### Verification checklist
- [ ] Unauthorized `v:` registration rejected on every node.
- [ ] A higher-"weight" fabricated fork that violates a signed checkpoint is rejected by `reorg_to_branch`.
- [ ] Succession node refuses a snapshot+headers not descending from a checkpoint ≥ snapshot height.
- [ ] Weight is continuous across a MINT→SETTLEMENT switchover (no zero-weight window) — simulate the migration.
- [ ] A settlement declaring `constituent_count > MAX_CONSTITUENTS_PER_SETTLEMENT` is rejected before it can contribute weight.

# PHASE 3 — Batched settlement (SETTLEMENT_MINT, recompute-authoritative)

Purpose: ~100–1000× fewer on-chain txs. **Additive** new TxType (does not remove MINT yet). Deploy-all-before-any-emit.

### Design decision baked in (C4): **recompute-authoritative, NO declared aggregate.**
The settlement carries **only** `{serving_node_id, epoch_id, constituents_merkle_root, constituent_count, node_signature}` (+ bound wallet from H6). It does **NOT** carry a declared `{recipient,amount}` vector or `total_burn`. Every node **recomputes** the authoritative amounts from replicated state in canonical order and credits that. This deletes the unsatisfiable equality gate (C4), makes per-leaf skip natural (M5), makes cap admission natural (M6), and shrinks the permanent record.

### Wire formats
- **TxType::SETTLEMENT_MINT = 0x70.** On-chain body (compact, permanent):
  `u8 type | Hash256 serving_node_id | Address(20) serving_node_wallet | u64 epoch_id | Hash256 constituents_merkle_root | u32 constituent_count | Sig64 node_signature` (node signs everything above).
- **MC_SETTLE_BODY blob** (companion, content-addressed by `constituents_merkle_root`): the fat `PlayEntry[]` (the PlayProof fields `check_play`/`compute_mint_outputs` need). Flooded on a new object channel, min-merged/deduped, **matured in the same 30s window** so every node holds it before the tx is mintable. **Prunable** after `FINALITY_DEPTH` (the storage win).

### Data structures
- `accplay:<epoch>:<session_id>` (node-local scratch) : per-play compact `PlayEntry` awaiting settlement; **kept until FINALITY_DEPTH** past confirmation, not deleted on first confirm (M1).
- `u:<session_id>` (existing) : per-constituent replay marker.
- `us:<serving_node_id>:<epoch_id>` (new) : prevents a node re-settling an epoch.
- `sb:<merkle_root>` : stored settle-body.

### Edits
- `src/core/transaction.h/.cpp` : add `TxType::SETTLEMENT_MINT`, `struct PlayEntry` (with canonical (re)serialize for Merkle leaves), `struct SettlementMintTx` (serialize/deserialize/tx_hash), `Transaction::from_settlement/parse_settlement` (mirror from_mint:540).
- `src/api/server.cpp` : **accrual** — at session.complete append `PlayEntry` to `accplay:<floor(block_ts/EPOCH_MS)>:<session_id>` (keep proof build+sign + anti-farm). New **epoch-close worker**: closes on **LOCAL wall clock** (emit timing is explicitly non-consensus — this dodges the empty-blocks-disabled deadlock, H7). On close: load epoch PlayEntries, sort canonically by `(content_hash, session_id)`, build `merkle_root_bytes` (core/merkle.cpp:25), emit ONE `SettlementMintTx` + flood the `MC_SETTLE_BODY`. Jitter emit by `hash(node_id) mod EPOCH_MS` (M2).
- `src/tokens/mint.cpp` : add `validate_and_settle(tx, body, db) -> {per_recipient_map, total_burn}` : verify node sig; recompute merkle root over body == committed root; for EACH constituent run `check_play`; then **recompute money** in canonical `(content_hash, session_id)` order — resolve SongSection **from STATE only** (H2, see Phase 4 requirement that all fields live in state), read live `play_count`, call `compute_mint_outputs`, accumulate, **step `s:` counter per constituent**, add post-10k burn. **Per-leaf skip** (deterministic, committed in outcome): a constituent that is already `u:`-used, or whose listener balance < recomputed burn (M5), or that would cross `SUPPLY_CAP` (M6), is **skipped** — recompute simply omits it; the walk continues. Weight counts only **credited** constituents.
- `src/core/chain.cpp : apply dispatch (217) + new apply_settlement_mint` : load body by root; call `validate_and_settle`; apply the recomputed map via `credit_many`, debit listener burns, `SUPPLY_CAP` guard on the running sum, write `u:` per credited constituent + `us:` epoch marker. Extend `first_unappliable_tx` (candidate.cpp:489) so a settlement **missing its body** is treated not-yet-appliable (dropped, never wedges the block).
- `src/api/rats_api.cpp` : `tx_preflight_ok` SETTLEMENT case = structural deserialize + node sig + **body-present-for-this-root** gate. New `MC_SETTLE_BODY` handler: dedup by root, **verify root before storing** and **cap `constituent_count` + body bytes** (M7 DoS), wake candidates. **Pre-verify each constituent ECDSA at body ingest and cache the verdict keyed by leaf hash** (H5) so apply never re-verifies.
- `src/storage/database.{h,cpp}` : `accplay:` put/iterate/del, `sb:` get/put by root + prune-after-FINALITY hook, leaf-verdict cache.

### Consensus predicate (preflight == block-apply)
```
valid_settlement(tx, db):
  node_sig(tx) verifies vs v:[tx.serving_node_id].pubkey
  AND tx.serving_node_wallet == address(v:[serving_node_id].pubkey)     # H6
  AND tx.constituent_count <= MAX_CONSTITUENTS_PER_SETTLEMENT           # C2/M7
  AND body(tx.merkle_root) present
  AND merkle_root_bytes(body) == tx.merkle_root
  AND us:[serving_node_id:epoch_id] absent
apply is authoritative: credited set + amounts + burn are RECOMPUTED, not declared.
  For each constituent in canonical (content_hash, session_id) order:
    if u:[session]==used OR listener_bal<burn OR supply+amt>SUPPLY_CAP: SKIP
    else credit compute_mint_outputs(...), step s:counter, write u:[session]
```
There is **no declared-aggregate equality check** — recompute is the sole source of truth, identical on every node because it walks a fixed canonical order over replicated inputs.

### Migration within Phase 3
Ship the binary to **every node before any node emits** a settlement (verify-before-emit). Run per-play MINT + settlement in parallel; migrate emit to settlement; **retire per-play MINT emit** only after weight-continuity (Phase 2C) is confirmed live.

### Verification checklist
- [ ] Two nodes with different LevelDB iteration/compaction order produce byte-identical credited maps from the same settlement+body.
- [ ] A settlement whose body a node lacks is dropped (not wedged); body arrives → next block includes it.
- [ ] Post-10k tier boundary: concurrent plays of one song across two nodes get identical ordinals/tiers regardless of settlement arrival order (recompute steps `s:` deterministically).
- [ ] Drained-listener constituent is skipped identically on all nodes; the rest of the epoch still credits.
- [ ] Reorg after first-confirm does NOT lose plays (accrual kept to `FINALITY_DEPTH`).
- [ ] Oversized/root-mismatched `MC_SETTLE_BODY` rejected before storage.
- [ ] Constituent ECDSA verified at body-ingest; apply path performs **zero** constituent re-verifies (measure).

# PHASE 4 — state_root + snapshots + pruning (succession)

Purpose: commit to full ledger state, catch any silent per-node drift, and let a new operator sync from snapshot + recent blocks. **Consensus-breaking header change** (folds into the flag-day format).

### state_root (Subsystem 3)
- `src/core/block.h : BLOCK_VERSION (29) + BlockHeader (~166)` : add `Hash256 state_root`; bump `BLOCK_VERSION 3→4`.
- `src/core/block.cpp : header serialize (68-79) / deserialize (138-140)` : write/read `state_root` after `timestamp_ms` so it enters the hash preimage.
- `src/storage/database.{h,cpp}` : maintain an **LtHash16** additive lattice accumulator (1024×uint16, component-wise add mod 2^16 over `Blake2b(key||0x00||value)`; `state_root = SHA256(limbs)`). **MUST be additive, not XOR** (open-risk). Hook it on writes to an **explicit allowlist of STATE prefixes** (M3) — NOT a denylist: `a: nv: c:total_supply s: f: fph: i: sm: ia: ig: bh: u: v: founder: mlvl: mpub: mact: m: slashed: un: addrun: label: art_label: prop: propstatus: propvote: ha: hb: ht: d: fr: ml: ms:`. **Exclude** history (`b: k: h: cw: n: t:tip`), mempool (`p: pt: pv: pm: sp:`), and node-local (`ddur: ddaymax: accplay: sb:` + verdict caches). Per mutation: subtract old leaf, add new leaf.
- `src/core/chain.cpp : connect_block (75-168) + candidate producer` : after apply, compute accumulator digest; **producer stamps** `header.state_root`; **every validator recomputes and REJECTS** if `!= header.state_root`. This turns silent per-node balance drift into an instantly-detected consensus fault.
- **Prerequisite (H2):** every field `compute_mint_outputs` reads (artist_address, royalty_splits, duration, play_count) MUST live in a committed STATE prefix (`s:`/`sm:`), and `validate_and_settle` reads **only STATE, never a block body** — else pruned lightweight nodes fork from archival nodes.

### Snapshots + succession (Subsystem 3)
- `src/sync/snapshot.{h,cpp} (new) + rats_api.cpp` : every `SNAPSHOT_INTERVAL` blocks, at a height `< tip - FINALITY_DEPTH`, dump all STATE prefixes in **sorted-key order** into zstd chunks; manifest `{snapshot_height, block_hash, state_root, chunk_merkle_root, chunk_hashes[], total_supply, entry_count}` (content-addressed, byte-identical cross-node). RPCs: `snapshot.manifest`, `snapshot.chunk`, `headers.range`.
- Succession driver: pull **headers** genesis/checkpoint→tip (validate linkage + **signed checkpoints** from Phase 2B + weight); pick highest snapshot `≤ tip-FINALITY_DEPTH`; fetch+verify chunks vs manifest; reconstruct STATE; recompute LtHash; require `== manifest.state_root == validated header.state_root at that height`; **require the snapshot descends from a checkpoint newer than snapshot_height** (C5); then `connect_block` the tail. Never fetches a body below the snapshot.

### Pruning (Subsystem 3)
- MUST-KEEP: all STATE, all headers + `n:/h:/cw:`, recent bodies within `max(FINALITY_DEPTH, SERVE_DEPTH)`, latest snapshot, `t:tip`.
- PRUNABLE: bodies `b:` + `k:` below `PRUNE_DEPTH` (> `FINALITY_DEPTH`), superseded snapshots, `MC_SETTLE_BODY` past `FINALITY_DEPTH`, `u:` markers older than `SESSION_VALIDITY_BLOCKS`.
- **Replay-after-prune fix (H4):** a constituent's mint validity is bound to a **chain-anchored time** — the `PlayEntry` must reference a recent block hash whose height is within `SESSION_VALIDITY_BLOCKS`; `check_play` rejects constituents older than the window by that **anchored** height, not by node-claimed timestamp. `u:` markers are kept strictly `≥ SESSION_VALIDITY_BLOCKS`, so a pruned `u:` can never be re-minted (its anchor is already out of window). `SESSION_VALIDITY_BLOCKS ≥ PROPAGATION_WINDOW + settlement epoch`.
- `src/consensus/candidate.h` : define `PRUNE_DEPTH`, `SNAPSHOT_INTERVAL`, `SERVE_DEPTH`, `SESSION_VALIDITY_BLOCKS`.

### Consensus predicate
```
connect_ok(block): apply_ok(block) AND recompute_LtHash(state) == block.header.state_root
snapshot_trust(manifest): recompute_LtHash(imported) == manifest.state_root
                          == validated_header[manifest.height].state_root
                          AND descends_from_checkpoint(newer_than manifest.height)
mint_time_valid(constituent): anchor_height within [tip-SESSION_VALIDITY_BLOCKS, tip]
```

### Verification checklist
- [ ] Live-built node and rebuild-from-scratch node produce identical `state_root` at the same height (depends on Phase 0 rebuild fix — re-run).
- [ ] Node-local rows (`ddur:` etc.) changing does NOT change `state_root`.
- [ ] A deliberately drifted balance on one node makes its block fail to connect network-wide.
- [ ] Succession node syncs from snapshot+headers+tail to a `state_root` identical to a full-history node.
- [ ] Snapshot NOT descending from a fresh checkpoint is refused.
- [ ] Pruned node and archival node accept the identical settlement set (SongSection read from STATE only).
- [ ] A play whose anchor is older than `SESSION_VALIDITY_BLOCKS` is rejected even after its `u:` marker is pruned.
- [ ] `BlockPropagator` chunking handles the new header size.

# PHASE 5 — Two-tier mempool + verify-once + capped inclusion (throughput)

Purpose: sustain 3,472/s avg, ~15k/s peak. Block-validity **caps** are consensus-critical; the verdict cache is a local optimization.

### Wire / block params
- `src/core/block.h (29,35)` : `MAX_BLOCK_SIZE 2→8 MiB`; add `MAX_TXS_PER_BLOCK = 16384`. (Folds into the v4 flag-day format — no separate break.)
- `src/core/block.cpp : Block::validate` : **ENFORCE** reject if `transactions.size() > MAX_TXS_PER_BLOCK` or serialized size `> MAX_BLOCK_SIZE`, so an over-cap block is invalid network-wide (no cap-fork).

### Data structures
- `pv:<tx_hash>` verdict cache (29 B `{sender(20)|nonce(8)|flags(1)}`) — content-addressed "signature already verified."
- `pm:<submit_ms BE(8)><tx_hash>` maturity index for O(log N + K) oldest-mature seek.
- In-memory **HOT** mempool: bounded `H_max=131072`, ordered `(submit_ms, tx_hash)`, inline bytes + `{sender,nonce,size}`.

### Edits
- `src/api/rats_api.cpp : ingest_tx (3347)` : on `tx_preflight_ok` success, additionally write `pv:` + `pm:`. Single verify point. On min-merge `submit_ms` lowering, move the `pm:` key (bounded to one lowering per tx_hash from Phase 0).
- `src/storage/database.{h,cpp} (301-346)` : `pv:`/`pm:` put/get/del, `seek_mature(from_submit_ms)`, `pending_tx_count()`, extend `del_pending_tx` to drop `pv:`/`pm:`.
- `src/consensus/candidate.{h,cpp} (269, 314-427, 456-469)` : new bounded HOT index with a promotion step off `pm:` (local wall clock decides **when** to promote, never reaches block bytes). Replace `get_all_pending_txs` full-scan + per-tx deserialize+verify with HOT reads consulting `pv:` (extract sender/nonce only, **no re-verify**). **Capped selection**: lowest-`(submit_ms, tx_hash)` up to `MAX_TXS_PER_BLOCK`/`MAX_BLOCK_SIZE`; `block_ts = max included submit_ms`; then sort body by `(sender, nonce, tx_hash)`; existing `first_unappliable_tx` drops gaps. Add a **~1s tx-drain timer** distinct from the 5-min heartbeat poll. Deterministic cold expiry: drop `submit_ms < tip.block_ts - MEMPOOL_MAX_AGE_MS`.
- `src/core/chain.cpp : apply_transactions (199-276)` : before each `verify_signature`, look up `pv:`: **HIT → skip ECDSA**, run stateful checks only; **MISS** (peer-block tx never ingested) → full verify then populate `pv:`. So peer txs are always fully verified; self-built blocks are verify-free.
- `src/core/chain.cpp : connect_block drain (154-159)` : in the same WriteBatch that drops `p:/pt:`, also delete `pv:/pm:` and evict from HOT (no crash window).

### Consensus predicate (the caps are the consensus part)
```
block_valid_caps(block): tx_count <= MAX_TXS_PER_BLOCK AND size <= MAX_BLOCK_SIZE
selection: the lowest-(submit_ms, tx_hash) mature txs up to the caps
block_ts = max(included submit_ms)          # replicated value, no local clock
body order = (sender, nonce, tx_hash)
```
**Verdict-cache soundness:** `pv:` keyed by `tx_hash = sha256(raw)`; a cache hit skips re-verify only for bytes already verified at ingress (min-merge never mutates bytes); a MISS is fully verified. No unverified tx is ever applied. **Invariant/assert:** no code path writes `p:` without `pv:` (bootstrap/replay must repopulate or bypass `pv:` via full verify).

### Throughput safeguards
- Drain 16,384/s > 15k peak (headroom ~9%); sustained >16,384/s grows cold disk until `MEMPOOL_MAX_AGE_MS` expiry — **monitored disk ceiling + 500ms-block release valve** (M8).
- Settlement emit jitter (Phase 3, M2) keeps epoch bursts off the cap cliff.
- `u:` write volume (M9): with recompute-authoritative per-epoch settlement, consider a **per-epoch committed session-set** (bitmap/Merkle under the `us:` marker) instead of 300M `u:` point keys — deterministic cross-node dedup preserved. Track as a Phase-5 follow-up if `u:` write volume is measured as the wall.

# BOPWIRE TARGET-SYSTEM RED TEAM

Verified in tree: `count_plays` (chain.cpp:41) = number of `TxType::MINT` txs; fork choice = weight→height→timestamp→hash with weight = cumulative plays (chain.cpp:104,1166). `validate_mint` (mint.cpp:129) checks only duration≥30s, heartbeat≥dur/35000, and node sig vs `v:` registry. `compute_burn_rate` (ledger.cpp:10) is floating-point. `apply_mint` returns false on cap/insufficient-burn-balance (chain.cpp:300,308). `ddur:`/`ddaymax:` are written in the SAME WriteBatch as the mint (server.cpp:1076).

## CRITICAL — these invalidate the design as written

**C1 — Forge-a-play IS a chain-takeover, not just inflation. [Sub 1+4, forge(a)+divergence(b)+snapshot(e)]**
Fork weight = cumulative MINT/play count (verified chain.cpp:41,104; fork choice chain.cpp:1163-1173). Block production is costless: no PoW, no stake, no vote. So a party that can mint plays cheaply also manufactures fork weight cheaply, and the heaviest chain wins reorg (`reorg_to_branch`, chain.cpp:1159, can rewrite arbitrarily deep — no finality cap exists yet). Because a registered node can fabricate plays at will (see C3), forging is simultaneously money-printing AND the ability to out-weight and rewrite the honest chain, and to feed a fabricated history to any syncing/succession node (C5). Model-1 determinism guarantees honest nodes *agree given the same inputs*; it provides ZERO Sybil/alternate-history resistance. Mitigation: fork weight must be made unforgeable and decoupled from self-attested plays — either (a) signed checkpoints every K blocks from founder/mod keys (`founder:`/`mpub:` already exist) that hard-bound reorg depth and pin sync, plus (b) a reorg depth cap (FINALITY_DEPTH) enforced in `reorg_to_branch`. Without signed finality, nothing in Model 1 stops a costless heavier fork.

**C2 — Migrating per-play MINT → SETTLEMENT_MINT sets honest fork weight to ~0. [Sub 4, divergence(b)]**
Verified: `count_plays` counts only `TxType::MINT` (0x?), while SettlementMintTx is a NEW type 0x70. Subsystem 4 explicitly "retires per-play MINT." The instant that happens, honest blocks contain no MINT txs, so `count_plays`→0 and cumulative weight stops growing. Any node emitting even one legacy per-play MINT, or an attacker spamming them, produces a heavier chain and reorgs everyone. If you instead make weight count settlement *constituents*, an attacker forges a settlement declaring a huge constituent count (bounded only by the body cap) for near-unbounded weight. Mitigation: redefine weight to count verified constituents, cap constituents/settlement hard, and gate the whole thing behind signed checkpoints (C1). This edit is mandatory and currently absent from the Subsystem-4 edit list.

**C3 — The node signature is the ONLY thing standing between a registrant and unlimited mint. [Sub 1/4, forge(a)]**
`validate_mint` (mint.cpp:129-174) authenticates that *a registered node signed the proof*; it verifies nothing about a real listener, real audio delivery, or a real session. `session_id`, `player_address`, `player`/`seeder`/`mini` addresses, `total_duration_ms`, `heartbeat_count` are all node-chosen fields the node signs (server.cpp:992-1032). A malicious registered node loops: fresh random `session_id`, duration=30000, heartbeat=1, player/seeder/mini = wallets it controls → mints node(1)+discoverer(1)+seeder(1)+mini(1) spendable tokens/fake play (mint.cpp:68-124), escrow lane moderator-gated. The anti-farm per-device caps (recent commits, server.cpp `ddur:`) are node-local, NON-consensus, and a malicious node simply doesn't run them. So the entire economy's integrity == the `v:` registration gate. I could not find any stake/permission on `v:` writes; if registration is open, forge-inflate is unbounded and C1/C2 make it a takeover. Mitigation: `v:` registration MUST be permissioned/staked (slashable), OR plays must carry a listener-device signature (proof-of-listener) so the node cannot unilaterally fabricate the counterparty. This is the deepest hole; batching neither creates nor closes it, but batching + play-weighted fork choice amplify it.

**C4 — The "declared aggregate == recompute, else reject" rule is unsatisfiable for anything supply- or tier-dependent → settlement liveness collapse. [Sub 1, divergence(b)+liveness(d)]**
Both `play_count` (tier) and `burn_amount` (via `compute_burn_rate(total_supply)`, ledger.cpp:10, mint.cpp:74-81) are functions of GLOBAL MUTABLE state that other settlements in the same/earlier blocks move. A node building its epoch settlement cannot know the block-apply order of every other node's concurrent settlements, so it cannot predict the `play_count` ordinal (tier) or the `total_supply` (burn) that recompute will assign. Post-10k this bites EVERY settlement, because every mint/burn shifts `total_supply`, so every post-10k burn declaration depends on the global ordering the declarer can't see → deterministic reject → rebuild → reject again under churn. The design lists this only as an "open risk / consider." It is fatal, not optional. Decisive fix: DROP the declared `{recipient,amount}` vector and the equality gate entirely. Make settlement carry only `{serving_node_id, epoch, constituents_merkle_root, node_sig}`; every node RECOMPUTES the authoritative amounts from replicated state in canonical order and credits that. Recompute is already the source of truth, the declared vector is pure redundant attack surface, and removing it also shrinks the permanent on-chain record (helps the storage goal). This single change also resolves M5 (per-leaf skip) and H4 dedup cleanly.

**C5 — Trustless snapshot sync reduces entirely to checkpoint freshness, which is currently just genesis. [Sub 3, snapshot(e)]**
The succession flow validates snapshot chunks against `manifest.state_root` and pins that against the header chain's `state_root`. But header-chain validity rests only on "prev_hash linkage + checkpoints + fork weight." With weight costless and forgeable (C1/C2) and checkpoints = hardcoded genesis, an attacker fabricates a higher-weight header chain terminating in a fabricated-but-internally-consistent `state_root`, serves a matching snapshot, and the new/pruned-then-resyncing node adopts entirely attacker-controlled balances. The LtHash accumulator proves internal consistency, NOT authenticity. Mitigation (shared with C1): periodic founder/mod-signed checkpoints published on-chain and hardcoded/rotated in the binary; a syncing node must refuse any snapshot/header chain not descending from a checkpoint newer than the snapshot height.

## HIGH

**H1 — Floating-point burn breaks Model-1 determinism. [Sub 1/3, divergence(b)]**
`compute_burn_rate` computes `pct*pct*pct*1000.0*TOKEN_DECIMALS` in `double` (verified ledger.cpp:14-20). IEEE `double` results are not guaranteed bit-identical across compilers, `-ffast-math`, x86 vs ARM, or FPU vs SSE rounding. Under state_root (Subsystem 3) any one-ULP divergence in a single post-10k burn makes two honest nodes compute different `total_supply`, different accumulator, different `state_root` → block fails to connect network-wide → chain wedges. Mitigation: replace with integer-only fixed-point arithmetic (integer cubic over scaled ints). Must land BEFORE state_root is consensus-load-bearing.

**H2 — Recompute reads SongSection from the block BODY, which Subsystem 3 prunes → pruned/archival fork. [Sub 1 vs Sub 3, divergence(b)+snapshot(e)]**
Step 5(d) resolves SongSection via `get_content_height→get_block→song` — a historical BODY read. Subsystem 3 deletes bodies below PRUNE_DEPTH. A settlement referencing a song whose registration body is pruned becomes un-recomputable on lightweight nodes → they reject it while archival nodes accept → deterministic fork by node role. Mitigation: the full SongSection needed by `compute_mint_outputs` (artist_address, royalty_splits, duration, play_count) MUST live in STATE (e.g. `sm:`/`s:` under the state-root set) and recompute must read only STATE, never a body. Verify every field `compute_mint_outputs` touches is in the committed state set.

**H3 — Origin-chosen `submit_ms` is backdatable → maturity before propagation → data-availability fork + re-flood amplification. [Sub 1/2, divergence(b)+liveness(d)]**
Maturity = `block_ts - submit_ms ≥ 30s`, and `submit_ms` is stamped by the origin. Backdating `submit_ms` 30s into the past makes a tx/settlement "mature" the instant it is created, before peers receive it. Only the origin can then include it; peers build a different block → fork resolved by fork choice (and a backdated timestamp can win the timestamp tiebreak). Separately, min-merge re-floods whenever a lower `submit_ms` arrives, so an attacker re-broadcasting the same tx_hash with monotonically decreasing `submit_ms` forces unbounded re-flood + `pm:` reindex (DoS). Mitigation: reject any tx whose `submit_ms < tip.block_ts − PROPAGATION_WINDOW_MS` (and > tip + small skew); cap re-flood to one lowering per tx_hash; consider gating maturity additionally on a bounded local first-seen time (as a liveness floor, not in block bytes).

**H4 — Replay-after-prune double-mint. [Sub 1/3, forge(a)+double-credit(c)]**
`u:`+session_id is the cross-node replay guard (verified chain.cpp:320, mint.cpp:133). Subsystem 3 makes `u:` prunable after SESSION_VALIDITY_BLOCKS. Since `session_id` and `play_end_timestamp` are node-chosen and node-signed, a node re-submits an old play with a *fresh* timestamp and a session_id whose `u:` marker was pruned → passes is_session_used → double-mint. Mitigation: bind mint validity to a chain-anchored time (e.g. constituent must reference a recent block hash whose height is within the validity window) AND keep `u:` markers strictly ≥ the validity window; reject constituents older than the window by that anchored time, not by node-claimed timestamp.

**H5 — Per-constituent ECDSA verify happens at APPLY, uncached, bursty at epoch close → block-production stall. [Sub 1/2, throughput(f)]**
The verify-once verdict cache (`pv:`) covers whole txs by tx_hash; it does NOT cover the per-constituent play signatures inside a settlement, which `apply_settlement_mint` verifies during `connect_block` on every node. Settlements land in bursts at epoch boundaries, so a single block can carry tens of thousands of constituent ECDSA verifies executed synchronously in the apply path (~40µs each → 10k = 0.4s of pure verify inside one block connect), starving 1s block production at peak. Mitigation: pre-verify constituents at MC_SETTLE_BODY ingest and cache verdicts keyed by leaf hash; apply consults the cache and never re-verifies. Otherwise the "verify-once" win doesn't apply to the dominant verify volume.

**H6 — serving-node WALLET isn't bound; `v:` holds pubkey+endpoint, not the wallet. [Sub 1, forge(a)+divergence(b)]**
`compute_mint_outputs` credits `serving_node_address` (the node WALLET, server.cpp:1036 passes `node_keypair_.address`), but the settlement tx carries `serving_node_id` and the `v:` registry value is `pubkey33+endpoint` (mint.cpp:161). Recompute on a validator must resolve the wallet deterministically; if the signing key ≠ wallet key, it cannot, and either misroutes the node share or diverges. Mitigation: commit the serving-node wallet address inside the settlement under the node signature (or make wallet == address(v:pubkey) an enforced invariant), and recompute from that single bound source.

**H7 — Epoch close gated on block_ts + empty blocks disabled = low-volume deadlock. [Sub 1/4, liveness(d)]**
Empty blocks are disabled (candidate.cpp:534). Settlements are the only txs the reward path produces, but the design closes epochs when `block.timestamp_ms` advances — which requires a block, which requires a tx, which is the settlement that won't emit until the epoch closes. Quiet periods / single-node bootstrap deadlock: accrued plays never settle. Mitigation: epoch-close/emit timing is explicitly non-consensus, so close on LOCAL wall clock (allowed here) and never gate emit on the chain clock; validity is recomputed regardless, so wall-clock bucketing is safe.

## MEDIUM

**M1 — Accrual scratch pruned on first-confirm → reorg loses plays permanently. [Sub 1, double-credit/lost(c)]** Design deletes `accplay:` "once the settlement confirms." If confirm = first inclusion and the block later reorgs out AFTER the local body/accrual are GC'd, the constituent list is unrecoverable and those plays never mint. Keep accrual + a local copy of the body until FINALITY_DEPTH.

**M2 — Synchronized epoch bursts vs the 16,384/1s cap. [Sub 2, throughput(f)]** If all nodes align epochs to wall-clock boundaries, every node emits at t≡0 mod EPOCH_MS → periodic spikes far above the 3,472/s average, hitting the cap cliff (drain 16,384/s is only ~9% over 15k peak). Jitter emit by `hash(node_id) mod EPOCH_MS`.

**M3 — Non-consensus rows in the consensus WriteBatch. [Sub 3, divergence(b)]** Verified: `ddur:`/`ddaymax:` are written in the same batch as the mint (server.cpp:1076-1078). If the LtHash accumulator hooks writes by prefix, these node-local rows must be EXPLICITLY excluded or every node's state_root diverges. Pin an exact allowlist of STATE prefixes, not a denylist.

**M4 — clear_derived_state rebuild incompleteness → rebuild root ≠ live root. [Sub 3, divergence/snapshot]** Per the design's own audit, `clear_derived_state` wipes only `a:/s:/f:/i:/u:/bh:` but nonces `nv:`, `c:total_supply`, governance/label/prop/moderation prefixes are also block-derived. If rebuild doesn't reconstruct EVERY state prefix identically, a resynced node's accumulator differs from a live node's → wedge. Full audit + fix required before state_root is trusted.

**M5 — Post-10k listener with drained balance fails the WHOLE settlement. [Sub 1, double-credit(c)+liveness(d)]** Verified: `apply_mint` returns false if `bal < burn_amount` (chain.cpp:308). Batched, one poor listener wedges a node's entire epoch. Fix: per-leaf skip with the skip determined by recompute (natural once C4's recompute-authoritative model is adopted — recompute simply omits unpayable/used leaves deterministically).

**M6 — SUPPLY_CAP guard rejects the summed batch wholesale near the cap. [Sub 1, liveness(d)]** `apply_mint` refuses if `supply+total>SUPPLY_CAP` (chain.cpp:300). A large settlement fails entirely near the cap, and cap-race ordering across nodes' settlements differs. Recompute-authoritative + per-leaf admission until the cap binds is the deterministic fix.

**M7 — MC_SETTLE_BODY DoS. [Sub 1, throughput(f)]** Malicious huge bodies or root-mismatched bodies. Cap constituent_count and body bytes; verify merkle root BEFORE storing; dedup by root.

**M8 — Cold-tier unbounded under sustained >16,384/s. [Sub 2, throughput(f)]** Drain barely exceeds peak; a prolonged spike grows cold disk until MEMPOOL_MAX_AGE expiry. Needs a monitored disk ceiling and a release-valve (500ms blocks).

**M9 — 300M `u:` writes/day re-inflate the write volume batching aimed to cut. [Sub 1, throughput(f)]** One `u:` per constituent restores per-play write amplification in the replay index. Consider a per-epoch committed session-set (bitmap/Merkle) rather than 300M point keys — but keep cross-node dedup deterministic.

**M10 — block_ts = max(submit_ms) clustering. [Sub 2, divergence(b)]** Under backlog many blocks share a submit_ms cluster; confirm `tip_is_better` timestamp tiebreak and any monotonicity assertion tolerate equal/near-equal adjacent timestamps.

**L1** SongSection (royalty_splits/artist) mutated between play and settlement → recompute uses live state → reject or economic drift (deterministic, liveness-only).

## Bottom line
The batching/mempool/pruning machinery (Subsystems 1–3) is internally coherent, but it is built on a consensus base (Subsystem 4 / Model 1) that has **no Sybil resistance and a forgeable, play-counted fork weight**. Fix order:
1. **C3 + C1/C2 first** — permission/stake `v:` (or add proof-of-listener) and add signed checkpoints + a reorg-depth cap, and redefine fork weight so batching doesn't zero it. Nothing else matters until forging a play is no longer both free money and free chain-takeover.
2. **C4** — make settlement recompute-authoritative; delete the declared-aggregate equality gate (also fixes M5/M6 and simplifies H4).
3. **H1 (integer burn), H2 (SongSection in state), H6 (bind node wallet), M3/M4 (state-set discipline)** — all prerequisites before state_root goes load-bearing.
4. **H3/H4/H7** — close the submit_ms backdating, replay-after-prune, and epoch-close deadlock.
5. **H5/M2/M9** — the real throughput cliffs at 15k/s (constituent verify at apply, synchronized bursts, replay-marker write volume), none of which the current verify-once/verdict-cache design actually covers.

## SUBSYSTEM DESIGNS

### Batched on-chain settlement mint (SettlementMintTx) — replaces the per-play direct LevelDB write at session.complete (server.cpp:1061) with a replicated, forge-verified, batched on-chain mint that all Model-1 nodes agree on without voting.
**Mechanism:** END-TO-END:

1) ACCRUAL (node-local, non-consensus). At session.complete the serving node still builds and node-signs the PlayProof exactly as today (server.cpp:990-1032), but instead of chain_.apply_mint + db_.write it appends the proof to a node-local epoch bucket and does NOT touch chain state. Two node-local LevelDB families: "accplay:"+epoch+":"+session_id -> serialized compact PlayEntry (the fields compute_mint_outputs/validate_mint need), and a running tally is unnecessary because the aggregate is derived at emit. Bucketing key epoch = play_end_timestamp / EPOCH_MS (EPOCH_MS ~ 10s). This store is scratch: it holds only the current few open epochs and is deleted once the epoch's settlement tx confirms.

2) EPOCH CLOSE + EMIT. An epoch closes when the node observes the chain's own clock advance past it: latest applied block.header.timestamp_ms >= (epoch+1)*EPOCH_MS + PROPAGATION_WINDOW_MS (a replicated value, not wall clock; even the emit trigger avoids a local-clock read, and emit timing is not consensus-critical anyway). On close the node loads all PlayEntries for that epoch, groups by content_hash, sorts constituents canonically by (content_hash, session_id), builds the Merkle root over their canonical serializations (reusing merkle_root_bytes, core/merkle.cpp:25), recomputes the per-recipient aggregate the SAME way a validator will (see step 5) using the song's current on-chain play_count, and emits ONE SettlementMintTx per (node, epoch) via RatsApi::ingest_tx — flooded + min-merged like every other tx.

3) WIRE. New TxType::SETTLEMENT_MINT = 0x70. On-chain (permanent, compact) body: u8 type | Hash256 serving_node_id | u64 epoch_id | Hash256 constituents_merkle_root | u32 recipient_count | recipient_count×{Address(20),u64 amount} (sorted by address asc — the declared aggregate) | u64 total_burn | u32 constituent_count | Sig64 node_signature (serving node signs everything above via its 'v:' key). The fat constituent PlayEntry list is NOT in the block — it is a companion content-addressed blob keyed by constituents_merkle_root, flooded as a new MC_SETTLE_BODY object, min-merged/deduped and matured in the same 30s window so every node holds it before the settlement is mintable. That blob is prunable (kept only by archival role), which is what delivers the storage win.

4) FLOOD/MATURITY. The settlement tx rides the existing librats MC_TX_TYPE flood + submit_ms min-merge (rats_api.cpp ingest_tx:3347) and the 30s PROPAGATION_WINDOW maturity gate in candidate.cpp:465 unchanged; the companion body floods on MC_SETTLE_BODY and is gated 'body present' in preflight so a settlement whose body a node hasn't received is treated as not-yet-appliable (dropped by the first_unappliable_tx trial-apply gate, candidate.cpp:489, never wedging the block).

5) FORGE CHECK WITHOUT RE-RUNNING PLAYS (every validating node, identical). apply_settlement_mint(): (a) look up serving-node pubkey via 'v:'+hex(serving_node_id) and verify node_signature over the header — authenticates the batch (same registry validate_mint uses, mint.cpp:156). (b) Load the companion body by constituents_merkle_root; recompute merkle_root_bytes over its entries and require == committed root (binds exactly which plays). (c) For EACH constituent PlayEntry run the existing per-play gate: node_signature over proof.sign_message() verifies against the SAME 'v:' pubkey, total_duration_ms>=30000, heartbeat plausibility (lift validate_mint mint.cpp:129-174 into a reusable check_play()). Nodes never re-stream audio or re-observe heartbeats — they verify the serving node's signature attesting each play, exactly the trust model of today's single MintTx, just batched. (d) Recompute the money: iterate constituents in canonical (content_hash, session_id) order; for each, resolve the on-chain SongSection via db_.get_content_height(content_hash)->get_block_hash->get_block->song (server.cpp:812-817 path), read the song's LIVE running play_count, call compute_mint_outputs(proof, song, play_count, node_id, serving_node_address), accumulate outputs into a per-address map, then step that song's counter (update_song_state) and, post-10k, add the recomputed burn for that play. (e) Require the accumulated per-recipient map == the tx's declared {recipient,amount} list byte-for-byte (both sorted by address) and accumulated burn == total_burn; reject the whole tx on mismatch (Model-1 byte-identical discipline). (f) Replay/double-credit: reject if any constituent session_id already carries a 'u:' marker or repeats within the batch, and reject if 'us:'+serving_node_id+':'+epoch already exists; on success write a 'u:'+session_id marker for every constituent plus the 'us:' epoch marker. (g) SUPPLY_CAP guard on the summed mint, burn debits from each listener, credit_many for the aggregate — reusing apply_mint's existing guards (chain.cpp:288-336).

**Determinism:** No voting, no producer/follower, no local clock in consensus — identical to Model 1. The credited amounts are a PURE FUNCTION of replicated inputs: the flooded+min-merged SettlementMintTx, its Merkle-committed companion body (both converged before maturity), and current on-chain state (per-song play_count counter, total_supply, SongSection). Every node applies settlements in the same block order and, WITHIN a settlement, walks constituents in a fixed canonical order (content_hash, then session_id), so each node assigns each play the SAME play_count ordinal and therefore the SAME tier and burn — regardless of how late the settlement lands relative to the play. The KEY to tier determinism-under-latency: the origin node never commits a play_count into the proof; the chain RE-DERIVES each play's ordinal from its own monotonic 's:' counter at apply time, stepping it per constituent exactly as the old per-play update_song_state did. Two nodes' concurrent plays of the same song get ordinals in the deterministic settlement-apply order, identical on every node. The block timestamp remains the song submit_ms or max tx submit_ms (candidate.cpp:507-541), never wall clock; epoch bucketing is a node-local batching choice that only affects which plays share a tx, not the math, and the tx's timing still rides origin-stamped submit_ms like every other tx. ### Two-tier mempool + high-throughput deterministic tx-only blocks (verify-once, content-addressed verdict cache, capped deterministic inclusion) for Bopwire Model 1 consensus, sized for 3,472 tx/s avg / 15,000 tx/s peak.
**Mechanism:** END-TO-END PIPELINE.

(1) INGRESS / verify-ONCE. A tx arrives on the MC_TX flood in RatsApi::ingest_tx (rats_api.cpp:3347). It runs tx_preflight_ok ONCE (deserialize + secp256k1 signature verify + structural rules). On success it does three writes instead of today's two: (a) p:<tx_hash> = raw bytes (unchanged, min-merged), (b) pt:<tx_hash> = submit_ms (unchanged semilattice value), (c) NEW pv:<tx_hash> = a fixed 29-byte verdict record {sender(20) | nonce(8) | flags(1)} — the content-addressed verdict cache, and (d) NEW pm:<submit_ms BE(8)><tx_hash> = empty — a maturity-ordered secondary index. Min-merge is unchanged: re-flood/re-store only when new-or-lower submit_ms; when submit_ms is lowered the old pm: key is deleted and re-inserted at the new position. Because tx_hash = sha256(raw), the verdict is a pure function of immutable bytes; the min-merge only ever changes the pt:/pm: metadata, never the bytes, so the cached verdict stays valid forever. This is the ONE and only signature verification for a self-ingested tx.

(2) COLD tier (LevelDB, durable). p:/pt:/pv:/pm: together are the cold buffer. It holds (a) IMMATURE txs still inside PROPAGATION_WINDOW_MS (30 s), which physically cannot be included yet, and (b) OVERFLOW: mature txs beyond the current block's capacity. It is min-merged, order-independent, and survives restart. The pm: index lets the producer seek the oldest mature txs in O(log N + K) instead of today's full get_all_pending_txs() O(N) scan (candidate.cpp:269).

(3) HOT tier (in-memory, bounded). A new Mempool object owns an in-memory index of MATURE, ready-to-include txs, ordered by (submit_ms, tx_hash), each carrying {tx_hash, submit_ms, sender, nonce, size, raw-bytes-inline}. A maturation step (folded into the producer wake/poll) walks pm: forward and PROMOTES any tx whose age >= 30 s into the hot index, up to a bound H_max = 131,072 entries (~40 MB with 300 B inline). Local wall-clock is read ONLY to decide maturity/promotion timing — it never reaches block bytes (identical to today's candidate.cpp:237 comment). If hot is full, promotion stops; the tx stays durable in cold (back-pressure, see below).

(4) BUILD — deterministic capped inclusion (replaces candidate.cpp:456-469). **Determinism:** All honest nodes derive the identical block from identical replicated inputs, no vote/producer/clock-read in the block bytes.

MEMBERSHIP: the selected set is the lowest-(submit_ms, tx_hash) C mature txs. submit_ms is the min-merged, origin-stamped, replicated value (semilattice invariant I1); tx_hash is a unique 32-byte content hash. So (submit_ms, tx_hash) is a TOTAL order over replicated values — every node ranks the mature pool identically and the first C are the same set everywhere.

CAP vs MATURITY-EDGE SKEW (the key argument): the cap takes the OLDEST C mature txs. Clock skew only perturbs membership at the maturity EDGE (txs right at age~30 s), i.e. the NEWEST mature txs — which are never in the oldest-C when the cap binds. TIMESTAMP: block_ts = MAX included submit_ms (or the song's submit_ms) — a replicated value taken from the deterministically-selected set, never a local clock read (invariant I4 preserved).

BODY ORDER: sort by (sender, nonce, tx_hash) — unique, total, identical everywhere (invariant I5); per-sender nonce monotonicity preserved; applicability gate deterministically drops gaps.

VERDICT CACHE SOUNDNESS: pv: is keyed by tx_hash = sha256(raw). Skipping re-verify on a cache hit is sound because the bytes that hash to tx_hash are exactly the bytes we verified at ingress; the min-merge never mutates bytes. A peer block carrying a tx we never ingested is a cache MISS and gets fully verified, so no unverified tx can ever be applied. Same bytes -> same hash -> same block -> agreement WITHOUT voting, exactly as Model 1 requires.

Hot-index fullness only throttles local promotion timing (a WHEN, not a WHAT); the durable cold row is untouched, so a node that briefly refused to promote still converges once it drains.

### Subsystem 3 — Pruning + committed state root + verifiable state snapshots for lightweight/succession nodes (Model 1, vote-free).
**Mechanism:** GOAL: give every node a deterministic commitment to the WHOLE ledger state so (a) all honest nodes agree on state (not just on block bytes), and (b) a brand-new operator can sync from (snapshot + recent blocks) without downloading ~27 TB/yr of tx bodies, and verify that snapshot trustlessly.

(1) CANONICAL STATE (the authoritative, replicated key set — everything that a tx/block can mutate; NOT history, NOT mempool). Grouped by existing LevelDB prefix:
  - Ledger: balances `a:`<addr>->u64; nonces `nv:`<addr>->u64; total supply `c:total_supply`->u64.
  - Song registry/state: `s:`<ch> (play_count/discoverer/first_play), fingerprint `f:`<ch>, reverse-fp `fph:`<fphash>, chroma buckets `i:`<bucket>, metadata `sm:`<ch>, artist idx `ia:`, genre idx `ig:`, content-height `bh:`<ch>.
  - Session replay markers `u:`<session_id> (bounded, see below).
  - Validator/serving-node registry `v:`<serving_node_id>->pubkey33+endpoint (read by validate_mint mint.cpp:156).
  - Governance: `founder:`, `mlvl:`, `mpub:`, `mact:`, mirror `m:`, `slashed:`<addr>.
  - Names/labels: `un:`, `addrun:`, `label:`, `art_label:`.
  - Proposals: `prop:`, `propstatus:`, `propvote:`.
  - Moderation: `ha:`/`hb:`/`ht:` (hidden artist/album/title), `d:` (deleted), `fr:` (forgery reports), mod-log `ml:`/`ms:`.
  EXCLUDED from state (never in the root): history `b:`/`k:`/`h:`/`cw:`/`n:`/`t:tip`; mempool `p:`/`pt:`/`sp:`. These are per-node/derivable and would break cross-node determinism if committed.

(2) DETERMINISTIC STATE ROOT committed in the header. Add `Hash256 state_root` to BlockHeader (v3->v4). state_root is the digest of a homomorphic multiset (lattice) accumulator over the live STATE set: acc = SUM over every live (key,value) of Blake2b(key||0x00||value) expanded into a fixed vector of 1024 uint16 limbs, combined by component-wise addition mod 2^16 (LtHash16, Folly-style). Committed value = SHA256(limb vector). Chosen over a Merkle-Patricia/bucketed-Merkle tree because at 15k tx/s ~40k distinct keys change per block, which dirties essentially every bucket of any tree (degrading to a full O(N) re-hash of millions of keys per block); the accumulator is O(changed keys): on each write we subtract Blake2b(key||old) and add Blake2b(key||new) (insert = add only, delete = subtract only). The operation is commutative + associative, so LevelDB write order and map-iteration order are irrelevant — every node lands on the same limb vector, hence the same state_root. This is exactly Model 1: identical replicated inputs -> identical bytes -> identical hash -> agreement, with no clock and no vote. Committing state_root ALSO closes the current reward-divergence hole: a node accepts a peer block only if its own recomputed accumulator equals the header's state_root, so the direct-write, per-node balance drift (server.cpp:1061/1081) can no longer stay hidden — blocks carrying divergent state simply fail to connect.

(3) SNAPSHOTS (amortized full-Merkle for inclusion proofs + trustless bulk sync). Every SNAPSHOT_INTERVAL blocks (heights ≡0 mod interval, chosen ~ a few hours of blocks) a node deterministically dumps all STATE prefixes in LevelDB sorted-key order into fixed-size chunks (zstd), and builds a manifest: {snapshot_height, snapshot_block_hash, state_root (copied from that height's header), chunk_merkle_root, [chunk hashes], total_supply, entry_count}. Because the dump is over a LevelDB read-snapshot at a fixed height and keys are sorted, the manifest is byte-identical on every node -> content-addressed by SHA256(manifest). The snapshot is only cut at a height already buried past FINALITY_DEPTH so it can never be reorged out.

(4) SUCCESSION / NEW-NODE SYNC (no trust):
  a. Pull HEADERS only from a hardcoded checkpoint (or genesis) to tip (~200 B each); validate prev_hash linkage + checkpoints + accumulate fork weight. Headers are the trust backbone.
  b. Pick highest snapshot_height ≤ (tip − FINALITY_DEPTH). Fetch manifest+chunks from peers (resumable).
  c. Verify each chunk vs manifest chunk-hash; reconstruct state into LevelDB; recompute the LtHash accumulator over every imported entry; require it == manifest.state_root AND manifest.state_root == the validated header.state_root at snapshot_height. Since the header chain is independently validated, the snapshot is cryptographically pinned — the provider cannot inject a fake balance.
  d. connect_block the tail (snapshot_height+1 .. tip) normally; each connect re-derives and re-checks state_root. Node is now fully verified without ever fetching a body below snapshot_height.

(5) PRUNING. MUST-KEEP: all STATE (above); ALL block headers + `n:`/`h:`/`cw:` height/weight indexes (tiny, needed for fork choice + checkpoints + trustless snapshot pinning); recent full block bodies within max(FINALITY_DEPTH, SERVE_DEPTH); the latest verified snapshot; `t:tip`. PRUNABLE: full block bodies `b:` and full-block checksums `k:` for height < PRUNE_DEPTH; superseded snapshots; `u:` session markers older than the session-validity window. Pruning = a background LevelDB range-delete over `b:`/`k:` for buried heights; headers stay.

(6) FOOTPRINT: history bodies are the wall (300M tx/day × 250 B ≈ 27 TB/yr, or far less after Subsystem-1 batched settlement collapses plays ~100–1000x). Pruning removes essentially all of it. Steady-state disk = live STATE (≈ one snapshot; scales with #accounts + #songs, NOT with tx count) + all headers (~GB/yr) + a bounded recent-body window. Audio is already off-chain, so a normal full node settles into tens of GB rather than unbounded TBs.

**Determinism:** Every node computes state_root as a pure function of the applied block set. The accumulator is a lattice/multiset hash: acc = Σ Blake2b(key||value) over live entries, combined by component-wise addition mod 2^16 across 1024 limbs. Addition is commutative and associative, so the result is independent of LevelDB write order, batch order, and std::map iteration order — the three things that differ between nodes. A per-key mutation is applied as (subtract old leaf, add new leaf), which is the same delta on every node given the same before/after values. No node reads a local clock (block timestamp_ms drives the session window; submit_ms drives tx maturity). No node votes: a peer block is accepted iff the receiver INDEPENDENTLY recomputes an accumulator equal to header.state_root, and the producer set the header from its own trial-apply. Identical replicated inputs -> identical limb vector -> identical SHA256 -> agreement. Snapshot verification is the same function run over the imported dump, checked against the same committed root, so bulk-sync inherits the identical determinism guarantee.

### Subsystem 4 — Migration + Phasing from single-founder off-chain-mint node to replicated, forge-verified, batched, prunable target — without dropping the live net.
**Mechanism:** END-TO-END PHASING (durability-first, each phase independently deployable + reversible; the one flag day is deliberately spent now while there are zero real users).

FACT BASELINE (verified in tree): genesis_writer.cpp mints block 1 with ONLY the founder GRANT and explicitly NO MintTx = no premine. So the ~800M "0x0c47E3" balance is NOT on-chain. It exists solely as non-consensus, per-node LevelDB credits written directly at session.complete (server.cpp:1061 apply_mint + :1081 db_.write). chain.cpp:1321 clear_derived_state() rebuilds ALL balances + total_supply from blocks on every resync — so those 800M are derived state that already vanishes on any rebuild and never replicated to any peer. from_mint (transaction.cpp:540) has zero callers, so nothing currently puts a mint on the wire.

WHAT TO DO WITH THE OFF-CHAIN 800M: ABANDON. Options weighed: (a) re-mint on-chain would require fabricating forge-valid MintTxs (node_signature over the v: registry) for sessions that never happened, and would inject 800M into a supply curve whose SUPPLY_FLOOR=1B/SUPPLY_CAP=2B burn economics start from that inflated base — poisons day one; (b) carry-forward is meaningless because the value was never real user value (founder test-farming) and is already destroyed by the next clear_derived_state; (c) abandon is free and automatic. Choose (c). The founder wallet resets to ~0 at cutover, which is correct.

PHASE 0 — DURABILITY (correctness beachhead, in-place on current chain, no format change, fully reversible). Turn the play reward into a real replicated, forge-verified transaction. session.complete stops doing the direct apply_mint+db_.write; instead it builds the MintTx it already builds, wraps it via the now-live Transaction::from_mint, stamps origin submit_ms ONCE, and hands it to ingest_tx (server.cpp path -> MC_TX_TYPE flood + min-merge mempool). Preflight (rats_api.cpp:3331, today a bare deserialize) and block-apply (chain.cpp:217-225, today no validation) BOTH gain: validate_mint (mint.cpp:129, node_signature vs v: registry + session-unused + duration/heartbeat) AND a recompute of compute_mint_outputs from on-chain SongSection + play_count + serving-node address, rejecting any MintTx whose outputs/burn differ. Now rewards ride the same order-independent min-merge mempool -> byte-identical blocks -> replicate to every node, forge-proof, no vanishing. Deployable to the single node with zero coordination; revert = restore direct write.

PHASE 1 — SCALE (batched settlement, SAME chain, ADDITIVE TxType, reversible). Add SettlementMintTx as a NEW TxType (does not replace MINT). Each serving node accrues per-recipient play credits into a node-local epoch accumulator plus the list of play leaves; epoch = floor(block_ts / EPOCH_MS) (a REPLICATED clock). When an epoch closes the node emits ONE SettlementMintTx = {epoch_id, serving_node_id, merkle_root(play leaves), aggregated per-recipient outputs, node_signature}. block-apply verifies node sig, epoch maturity, each leaf session-unused (reusing the existing "u:"+session_id marker), and that aggregated outputs == sum of per-leaf compute_mint_outputs; the merkle root lets a light/succession node prove one play's inclusion without the leaf set. ~100-1000x fewer on-chain txs. Ship binary to all nodes BEFORE any node emits (verify-before-emit), run per-play MINT + settlement in parallel, migrate emit to settlement, then retire per-play MINT. Additive -> old blocks still validate -> reversible.

PHASE 2 — STORAGE DURABILITY + SUCCESSION (state_root + snapshots). Add state_root to BlockHeader (BLOCK_VERSION bump). Producer computes it deterministically over the sorted consensus state tables (balances, song state, used-session markers, total_supply) after apply; every node recomputes and REJECTS a block whose state_root disagrees — silent per-node drift becomes an instantly-detected consensus fault. Then deterministic periodic state snapshots every N blocks: a NEW operator downloads the snapshot + recent blocks and verifies it against that block's state_root, syncing WITHOUT full history (lightweight + succession nodes). Archival full-history is an optional separate role. Snapshots are a pure sync optimization (no consensus change).

THE ONE FLAG DAY (chain wipe + re-bootstrap) — WARRANTED, spent now. Phase 2 bumps BLOCK_VERSION and Phase 1 adds a TxType; on a live chain those normally need activation-heights + permanent conditional code. Because there are no real users and genesis has no premine (nothing of value on-chain), build+test Phases 0-2 on a throwaway local chain, then do ONE wipe + re-bootstrap of the live VPS (wipe-and-run-home.ps1 locally; /var/lib/bopwire data dir + fresh genesis_writer on the VPS) onto the FINAL block version. That fold-in avoids carrying activation-height cruft forever. In-place would be chosen only if a user base existed to protect — it does not, so wipe is strictly cheaper. Sequence guarantee: the wipe happens LAST, after the mint path is proven, so the fresh chain is bootstrapped exactly once on the final format.

**Determinism:** No voting, no producer/follower, no wall clock in consensus — identical replicated inputs -> byte-identical blocks -> same hash -> agreement, exactly as Model 1 today. Per-play MINT (Phase 0): the MintTx floods via MC_TX_TYPE, mempool converges by min-merge on tx_hash keeping lower (submit_ms, hash), matures past PROPAGATION_WINDOW_MS=30s; every node recomputes outputs from (proof, on-chain SongSection, play_count, serving-node address) and rejects any mismatch, so the same tx set yields the same block. SettlementMintTx (Phase 1): epoch boundary = floor(block_timestamp_ms / EPOCH_MS) — derived from the REPLICATED block timestamp, never local time; the node-local accumulator/emit uses submit_ms/block_ts only to decide WHEN to emit, and the emitted tx is then validated purely from replicated data (leaves + on-chain state), so aggregation is a deterministic function every node re-derives identically. Leaves are content-addressed by session_id; the "u:"+session_id marker makes replay/double-settle deterministic across both mint paths. state_root (Phase 2) is recomputed over canonically-sorted consensus tables so it is iteration-order independent and identical on every node; a divergent node's block simply fails to connect. Node-local non-consensus rows (ddur:, ddaymax:, pending caches) are excluded from state_root and from the emitted tx, so they can differ per node without breaking agreement.
