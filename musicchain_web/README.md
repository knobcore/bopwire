# musicchain_web

Browser-based player for musicchain. The third client in the family
(alongside the Windows tracker and the Android player), built as a Vite
single-page app that talks to a mini-node over a WebSocket and does all
of its wallet / fingerprinting work in WebAssembly so no private key ever
leaves the browser.

## Architecture: mini-node as gateway

The browser does **not** speak the librats peer protocol directly. It
dials a mini-node (a long-lived VPS process) at

```
wss://<vps>:8082/
```

and the mini-node is the gateway onto the librats mesh — it does the
DHT lookups, the TCP peer connections, the UDP punch attempts, the
mDNS / UPnP probes, and proxies the resulting message stream back to
the browser as framed WebSocket payloads.

This is a hard architectural constraint, not a temporary stub. librats
fundamentally needs OS primitives that the browser sandbox does not
expose:

- listening TCP sockets (for inbound peer dials)
- pthreads with SharedArrayBuffer (librats' worker pool model doesn't
  map cleanly onto COOP/COEP-gated SAB)
- raw UDP sockets (for DHT and ICE)
- mDNS broadcast on the LAN
- UPnP IGD port-mapping
- netlink for interface enumeration

None of those are available to WebAssembly running under a `https://`
origin, and shimming them over WebRTC data channels would still need a
signalling server — which is the role the mini-node already plays.
See `C:\Users\lain\blockchain\musicchain\docs\mini_node_mesh.md` for
the wire protocol the mini-node speaks to the browser.

The WASM half mirrors the platform-specific glue used by the desktop and
Android builds for the parts that *do* belong in-browser: the same
libwally backend, the same BIP32 derivation path
(`m/44'/19779'/0'/0/0`), the same vendored Chromaprint 1.5.1 + KissFFT
— so a mnemonic produces the same address and a track produces the same
fingerprint on every platform.

## Prerequisites

- **emsdk** — Emscripten SDK, any recent `latest` tag. CMake's
  `find_package` machinery isn't enough; you need the `emcc` / `em++`
  compilers on PATH for `emcmake` and `emmake`.
- **Node.js** ≥ 20 (Vite 5 + ES2022 target).
- **CMake** ≥ 3.22.
- **Ninja** or **make** — whichever `emcmake cmake` picks up. The build
  script defers to the default generator, so installing either is fine.
- A POSIX shell for `build.sh`. On Windows that means Git Bash, MSYS,
  or WSL — the script uses `source`, `$(...)`, and `nproc`.

## One-time setup

```bash
# Emscripten SDK. build.sh looks for it at ~/emsdk first, then a couple
# of well-known fallbacks, then $EMSDK as an override.
git clone https://github.com/emscripten-core/emsdk.git ~/emsdk
cd ~/emsdk
./emsdk install latest
./emsdk activate latest
cd -

# Node deps.
cd musicchain_web
npm install
```

The chain core, libwally, and chromaprint sources are expected as
sibling directories (`../musicchain`, `../chromaprint-1.5.1`, etc.) —
the same checkout layout the desktop and Android builds use. The CMake
file resolves them relative to its own directory; no extra config knobs.
librats source is no longer consumed by this build — the mini-node
links it on the VPS side.

The prerequisites for the remaining WASM artifacts (wallet, chain,
chromaprint) are unchanged by the librats removal: same emsdk, same
Node version, same sibling-directory layout.

## Build the WASM

```bash
bash build.sh
```

That wipes `./build/`, runs `emcmake cmake -DCMAKE_BUILD_TYPE=Release ..`
and `emmake make -j<n>`, and drops the freshly-linked
`wasm/wallet.{js,wasm}`, `wasm/chromaprint.{js,wasm}`, and
`wasm/chain.{js,wasm}` next to the web app so the Vite dev server can
pick them up. No `rats.*` artifact is emitted — the `rats` CMake
target was removed and the browser reaches librats only through the
mini-node WebSocket.

After pulling a librats-removal change, do a hard reload in the
browser (Ctrl-Shift-R / Cmd-Shift-R). The network panel should show
*no* request for `/rats.js` or `/rats.wasm` — if you see one, you're
running stale bundled JS that still tries to dynamic-import the
WASM glue.

If `build.sh` exits with "EMSDK not found", either clone emsdk to the
default `~/emsdk` location or export `EMSDK=/path/to/emsdk` before
re-running.

There's also `npm run build:wasm` if you prefer to invoke the same step
through the package scripts.

## Run the dev server

```bash
npm install   # first time only
npm run dev
```

Vite serves the app at <http://localhost:5173>. The server is configured
with `strictPort: true`, so if 5173 is taken it'll fail loudly rather
than pick a random port. Two response headers are set for every request
(`Cross-Origin-Opener-Policy: same-origin` + `Cross-Origin-Embedder-Policy:
require-corp`) so that any future multi-threaded WASM (or any code path
that wants `SharedArrayBuffer`) just works without rewiring the dev
server.

The dev server hot-reloads TypeScript and CSS but does **not** re-run
`build.sh` — you have to re-invoke it manually after editing anything in
`native/` or under `../musicchain/`.

## Production build

```bash
npm run build
```

Outputs to `dist/`. The build uses `target: es2022` and emits
sourcemaps. The `wasm/` directory is wired in as Vite's `publicDir`, so
the static `.js` / `.wasm` artifacts produced by `build.sh` are copied
into `dist/` alongside the bundled app. Serve `dist/` from any static
host (the home node's own HTTP endpoint, an nginx, GitHub Pages — Vite
doesn't care).

Preview the production bundle locally with `npm run preview`.

## Pointing the player at a mini-node

The chain client reads its WebSocket endpoint from `localStorage` under
the key:

```
mc.node_url
```

When that key is missing, the player falls back to:

```
wss://<window.location.host>:8082/
```

— i.e. dial the mini-node gateway colocated with the host serving the
page on TCP/8082. Override in the browser console to point at a
different mini-node during development:

```js
localStorage.setItem('mc.node_url', 'wss://vps.example.com:8082/');
location.reload();
```

Clear with `localStorage.removeItem('mc.node_url')` to go back to the
same-origin default. See
`C:\Users\lain\blockchain\musicchain\docs\mini_node_mesh.md` for the
gateway protocol the mini-node speaks.

## How wallet.wasm is wired

The Emscripten link produces an ES module factory rather than a global
side-effect script. From `CMakeLists.txt`:

```
-sMODULARIZE=1
-sEXPORT_ES6=1
-sEXPORT_NAME=createWalletModule
-sENVIRONMENT=web,worker
```

so the JS side imports it as a default export and calls the factory to
get a fully-initialised module instance:

```ts
import createWalletModule from '/wallet.js';

const Module = await createWalletModule();
```

The factory `await`s WASM compilation, runs the runtime init, and
resolves once `_malloc` / `_free` and the `mc_web_*` entry points are
live. Each exported C symbol is bound to a typed JS function via
`Module.cwrap`:

```ts
// char* mc_web_bip39_generate_12(const uint8_t* entropy16);
const bip39Generate = Module.cwrap(
  'mc_web_bip39_generate_12', 'string', ['number']);

// void* mc_web_wallet_from_mnemonic(const char* mnemonic,
//                                   const char* passphrase);
const walletFromMnemonic = Module.cwrap(
  'mc_web_wallet_from_mnemonic', 'number', ['string', 'string']);

// char* mc_web_wallet_get_address(void* wallet);
const walletGetAddress = Module.cwrap(
  'mc_web_wallet_get_address', 'string', ['number']);

// void mc_web_wallet_free(void* wallet);
const walletFree = Module.cwrap(
  'mc_web_wallet_free', null, ['number']);
```

Conventions inherited from the Android / Windows builds:

- **Strings returned `string`** are read by Emscripten out of the WASM
  heap and copied into a JS string. The underlying C `malloc`'d buffer
  is leaked unless you take the `'number'` form and `Module._free()` it
  yourself; the `'string'` form is fine for the small outputs here
  (addresses, hex pubkeys, 12-word mnemonics).
- **Wallet handles are opaque `number`s** — heap pointers into the WASM
  module. Pass them straight back into `walletGetAddress`,
  `walletSign`, etc. Always pair `walletFromMnemonic` with
  `walletFree`.
- **Binary inputs** (entropy, signing payloads) go through `Module._malloc`
  + `Module.HEAPU8.set(...)` + the `'number'` cwrap signature. Free
  immediately after the call returns.

The exported symbol list in `CMakeLists.txt`
(`-sEXPORTED_FUNCTIONS=[...]`) is the single source of truth for what
the JS side can call. Anything not on that list is dead-code-eliminated
by `--closure=1 -O3`, so adding a new entry point means editing both
`native/wallet_glue.c` and the exported-functions list before
re-running `bash build.sh`.

The same factory pattern applies to `chromaprint.js`
(`createChromaprintModule`) and any future module agents add — keep the
`EXPORT_NAME` distinct so two modules can be loaded in the same page
without clobbering each other.
